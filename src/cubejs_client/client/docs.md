# Noridoc: client

Path: @/src/cubejs_client/client

### Overview
- The Cube API client: `CubeClient` (port of `CubeApi`'s sync half) and `AsyncCubeClient` (its async twin), plus the polling/retry state machine each is built on (`run_polling_loop`/`run_polling_loop_async`, port of `CubeApi.loadMethod`) and the continuous long-poll layered on top of it (`run_subscribe_loop`/`run_subscribe_loop_async`).
- Covers `load`/`meta`/`sql`/`dry_run`, plus `cube_sql`/`cube_sql_stream` (the CubeSQL interface) and `subscribe` (a degraded HTTP-polling `subscribe()`), with `subscription.py` supplying the cancellation handles.
- The integration point of the package — the only layer that imports from `core/`, `query/`, `transport/`, and `models/` all at once.

### How it fits into the larger codebase
- `__init__.py`'s `cube()` factory (@/src/cubejs_client/docs.md) constructs and returns a `CubeClient`; `AsyncCubeClient` is constructed directly (no factory) — these are the two ways users of the package obtain a client.
- Both clients take queries from @/src/cubejs_client/query/docs.md (`Query` or plain dict, normalized via `to_query_dict()`), send them through @/src/cubejs_client/transport/docs.md (`HttpTransport`/`AsyncHttpTransport` by default, or a caller-supplied `Transport`/`AsyncTransport`), decode responses with `core/response_decode.py`, and return @/src/cubejs_client/core/docs.md types (`ResultSet`, `Meta`) or `models/` types (`SqlQuery`, `ProgressResult`).

### Core Implementation
- `base.py` (`run_polling_loop` / `run_polling_loop_async`): a plain `while True` retry loop (the async version `await`s `request_fn()`/`sleep()` and additionally allows `progress_callback` to be either a sync function or a coroutine function) parameterized by a `request_fn` (called once per attempt) and a `to_result` callback (called once on a terminal, non-retryable response). Handles three retry triggers — HTTP 502 (bad gateway, retried unconditionally), a transport-reported "network error" (retried up to `network_error_retries` times), and a `"Continue wait"` error body (Cube's long-poll-in-progress signal, retried indefinitely with `progress_callback` invoked each time). Any other non-200 response raises `RequestError` (@/src/cubejs_client/errors.py). The two loops are hand-maintained twins, not shared code — a change to one's retry semantics must be mirrored in the other.
- `sync_client.py` (`CubeClient`) / `async_client.py` (`AsyncCubeClient`): constructs a default `HttpTransport`/`AsyncHttpTransport` from `api_url`/`api_token`/etc. unless a `transport` object is supplied directly. Each public method (`load`, `sql`, `meta`, `dry_run`) generates exactly **one** `baseRequestId` (a fresh UUID) before entering `_load_method` → `run_polling_loop`/`run_polling_loop_async`, and that same id is reused across every retry attempt of that logical request — only the transport-layer span counter increments per attempt (see @/src/cubejs_client/transport/docs.md).
- `load()` additionally runs `core.response_decode.decode_response_data()` on the raw response body (applying `cast_numerics` and unpacking compact/columnar formats) before constructing a `ResultSet`. Its `response_format="default"|"compact"|"columnar"` param is applied to the outgoing query via `patch_query_response_format()` (@/src/cubejs_client/query/docs.md), which adds `responseFormat` for the non-default formats; the response is then unpacked back to the row-dict shape by `decode_response_data()`.
- `cube_sql()`: POSTs a raw SQL string to `/cubesql` and returns `{schema, data, lastRefreshTime?}`, using `core/cubesql.py`'s `parse_cubesql_response` as `to_result` (the JSONL payload arrives inside the response body's `error` field). Runs through the same `_load_method`/polling loop as `load()`.
- `cube_sql_stream()`: a generator yielding typed chunks (`{"type": "schema"|"data"|"error", ...}`). It bypasses the polling loop entirely — it calls the transport's optional `request_stream()` (@/src/cubejs_client/transport/docs.md) directly and feeds the byte iterator through `core/cubesql.py`'s `iter_cubesql_chunks`/`iter_cubesql_chunks_async`. Raises `RuntimeError` if the transport has no `request_stream`.
- `subscribe()`: builds one poll request and runs `run_subscribe_loop`/`run_subscribe_loop_async` (see below), returning a `Subscription`/`AsyncSubscription` handle. The sync client runs the loop on a daemon `threading.Thread` (stopped by a `threading.Event`, with an interruptible `stop_event.wait()` as the sleep so `unsubscribe()` reacts promptly instead of waiting out `poll_interval`); the async client runs it as an `asyncio.Task` (stopped by task cancellation). The async `subscribe()` is deliberately *not* a coroutine — it schedules the task and returns the handle synchronously.
- `base.py` (`run_subscribe_loop` / `run_subscribe_loop_async`): the continuous long-poll. Each cycle refreshes authorization, runs one single-shot `run_polling_loop` pass, hands the result (or a caught `RequestError`) to the caller's `callback(error, result)`, then sleeps `poll_interval` and repeats until `should_stop()` flips. Errors delivered to the callback do **not** stop the loop (JS parity — JS calls `subscribeNext()` after an error too). The single-shot loops gained a `should_stop` predicate that raises the internal `_SubscriptionStopped` sentinel so a subscription can abort a mid-flight Continue-wait retry rather than waiting for it to terminate.
- `subscription.py` (`Subscription` / `AsyncSubscription`): the cancellation handles returned by `subscribe()` (replacing JS's `UnsubscribeObj`). `unsubscribe()` sets the stop event, joins the thread / cancels-and-awaits the task, and re-raises any exception the callback raised inside the loop (also stored on `.exception`).

### Things to Know
- The retry architecture is **deliberately restructured** from the JS SDK's closure/`ITransportResponse.subscribe()`-based re-invocation into a plain retry loop that calls `request_fn()` again per attempt. `subscribe()` layers a second loop (`run_subscribe_loop`) on top rather than reusing JS's re-invocation model.
- `subscribe()` is a **degraded HTTP-polling** port, not a push subscription — this port's transport has no push/`unsubscribe` channel, so it simply re-polls on `poll_interval`. Cancellation is **native** (a `threading.Event` for the sync client, task cancellation for the async one), which is why the JS SDK's `mutexObj`/`mutexKey` supersession machinery is **intentionally not ported**: `MutexChangedError` (@/src/cubejs_client/errors.py) is kept only as a name-compatible stub and is never raised. This is a locked design decision.
- Errors from a `subscribe()` poll are delivered to the callback and do **not** stop the loop (JS parity). The loop only stops when `unsubscribe()` is called, or when the callback itself raises — in which case the exception surfaces from `unsubscribe()`.
- `cube_sql`'s JS parser has `error === 'timeout'`/`'aborted'` sentinel branches; those are unreachable here because the polling loop raises `RequestError` first (see @/src/cubejs_client/core/docs.md).
- When `api_token` is a callable (not a string), `_update_authorization()` re-invokes it before every `_load_method` call and pushes the result onto `self._transport.authorization` — this is how token refresh is supported without the caller managing transport state directly. `AsyncCubeClient` calls the same callable synchronously (it is not itself awaited).
- `sql()`'s `to_result` branches on whether the response body is a list (blended/multi-query SQL) vs a single dict, returning `List[SqlQuery]` or a single `SqlQuery` respectively — callers must handle both return shapes.
- `async_client.py` and `sync_client.py` are not unified behind a shared base class; keeping them as separate, structurally-identical modules was a deliberate choice so each stays a straightforward, independently-readable port rather than introducing sync/async abstraction machinery for two call sites.
